# BE: Добавление общей настройки для отпусков — управление замещающими (даты и % замещения)

В системе отпусков отсутствует централизованная настройка, регулирующая правила добавления замещающих сотрудников.
Требуется ввести конфигурацию `LEAVE SUBSTITUTES MANAGEMENT` на уровне компании, которая позволит управлять:
- включением/отключением замещением;
- обязательностью указания дат замещения;
- ограничением максимального процента замещения.

Настройка должна быть реализована в рамках общей архитектуры `settings` и соответствовать ранее принятому REST-подходу.

#### Цель:

Реализовать расширяемую backend-конфигурацию для управления правилами назначения замещающих в модуле отпусков.

---
#### Функциональные требования:

**1. Модель настройки (Settings Options)**

Добавить в домен `settings.leave` новые параметры:
- `substitutionEnabled: boolean` — включено ли использование замещающих;
- `requireSubstitutionDates: boolean` — обязательность указания дат замещения;
- `maxSubstitutionPercent: integer` — максимальный допустимый процент замещения (1–100).

**2. REST API**

Использовать существующий endpoint:
```
GET /settings/leave
PUT /settings/leave
```

**Требуется:**
- Расширить DTO настроек отпусков новыми полями.
- Обеспечить обратную совместимость (новые поля опциональны для старых клиентов).
- Валидация на уровне API:
    - `maxSubstitutionPercent` ∈ [0, 100].

**3. Архитектурные требования**
- Реализация строго в домене:
```
settings.leave
```
- Не допускать хардкода — все ограничения берутся из настроек.
- Подготовить решение к расширению:
    - аналогичная логика может появиться в `absence`, `request`.

**4. Миграция и дефолтные значения**
- Для существующих клиентов:
    - `substitutionEnabled = false`
    - `requireSubstitutionDates = false`
    - `maxSubstitutionPercent = 100`
- Обеспечить корректную миграцию БД (nullable / default).

**5. Документация**
- Обновить OpenAPI/Swagger:
    - описание новых полей;
    - правила валидации.
- Добавить комментарии в DTO (JavaDoc)

**Дополнительно:**
- Покрыть бизнес-логику unit-тестами.

---
#### Критерии приемки:
- [ ] В API настроек отпусков доступны новые поля.
- [ ] Настройки корректно сохраняются и возвращаются через API.
- [ ] Серверная валидация:
    - запрещает добавление замещающих при выключенной настройке;
    - требует даты, если включена соответствующая настройка;
    - ограничивает процент замещения.
- [ ] Значения по умолчанию применяются для существующих данных.
- [ ] Решение не ломает текущие API-контракты.
- [ ] Swagger отражает изменения.
