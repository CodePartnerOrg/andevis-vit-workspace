# BE: Рефакторинг API настроек отпусков (package + URL) с приведением к REST-архитектуре

Текущая реализация настроек отпусков имеет несогласованную структуру пакетов и URL, что усложняет масштабирование и повторное использование для других доменов (заявления, отсутствия и т.д.).
Необходимо унифицировать API в соответствии с REST-подходом и заложить расширяемую структуру.

---
#### Цель:
Привести backend-слой (package structure + REST endpoints) к консистентной, расширяемой архитектуре для управления настройками различных модулей (leave, requests, absences и др.).

---
#### Требуется:

**1. Рефакторинг package structure**
Организовать пакеты по доменному принципу с возможностью масштабирования

Целевой подход:
```
ee.andevis.api.settings
    └── core
    ├── leave
    │     ├── controller
    │     ├── service
    │     ├── repository
    │     └── entity
    │     └── payload    
    ├── absence
    ├── request
    └── common
```

- Убрать текущую “плоскую” или несистемную структуру.
- Выделить общий модуль `settings` как агрегатор доменных настроек.
- Обеспечить единообразие именования.

**2. Рефакторинг REST API (URL)**
Привести endpoints к REST-конвенции с учётом расширяемости.
```
/settings/{domain}
```

**Примеры:**
- `/settings/core`
- `/settings/leave`
- `/settings/absence`
- `/settings/request`

**Для настроек отпусков:**
- `GET /settings/leave` — получение настроек
- `PUT /settings/leave` — обновление настроек

**Дополнительная иформация:**
- Исключить глаголы в URL (типа `/getSettings`, `/updateLeaveSettings`).
- Использовать HTTP-методы по назначению (GET, PUT). 
- Использовать стандарт REST API в названиях и методах https://restfulapi.net/resource-naming/ (обязательно внести требования по REST API документацию чтобы в последующих задачах новые модификации соотвествовали стандарту )
- Обеспечить консистентность с будущими модулями.


**3. Расширяемость**
Заложить принципы:
- Возможность добавления новых доменов без изменения существующих контрактов.
- Минимизация дублирования кода (использовать общий слой `common`).
- Унифицированный DTO/response format (где применимо).

**4. Обратная совместимость**
При необходимости предусмотреть временную поддержку старых endpoint (deprecated).

**5. Документация**
- Обновить OpenAPI/Swagger:
    - новые endpoints
    - описание структуры settings
- Отметить deprecated endpoints.

**6. Покрытие Unit tests**
Покрыть весь `ee.andevis.api.settings` unit тестами (mokito).

**Дополнительно (опционально):**
- Рассмотреть введение `SettingsFacade` или `SettingsService` для унификации доступа.

---
**Критерии приемки:**
- [ ] Все endpoints настроек отпусков соответствуют `/settings/leave`.
- [ ] Package structure приведена к доменной модели (`settings.leave.*`).
- [ ] API соответствует REST принципам (ресурсы, HTTP-методы).
- [ ] Добавление нового домена (например, absence) не требует рефакторинга существующего кода.
- [ ]  Swagger отражает актуальную структуру API.
- [ ] Старые endpoints либо удалены, либо помечены как deprecated.
