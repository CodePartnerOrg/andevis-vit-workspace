## Описание проблемы

В текущей реализации название и описание заявления хранятся только на одном языке. Система не поддерживает хранение переводов и не может автоматически отображать текст в зависимости от языка интерфейса пользователя.

Из-за этого для пользователей с разными языками интерфейса (ET/EN/RU) заявления отображаются некорректно или требуют создания отдельных заявлений для каждого языка.

## Описание задачи

### Модель данных

Расширить модель типов данных заявлений (application_properties) для хранения переводов:

* Название заявления
* Описание заявления (Lisainfo)

Поддерживаемые языки переводов:

* RU
* EN

Базовый язык системы:

* ET

### API и логика работы

Реализовать:

* сохранение переводов названия и описания
* обновление переводов
* получение текстов в зависимости от языка интерфейса пользователя

### Логика fallback

Если перевод отсутствует:

* система возвращает текст на базовом языке **ET**

### Использование

Логика должна корректно работать для:

* создания заявления
* просмотра заявления
* согласования заявления
* предпросмотра формы

---
### Технический аспект
Проанализировать как уже релизована логика переводов в references. 
---
## Acceptance Criteria

* В модели данных добавлена возможность хранения перевода **названия заявления (RU, EN)**
* В модели данных добавлена возможность хранения перевода **описания заявления (RU, EN)**
* API поддерживает сохранение переводов названия и описания
* API поддерживает обновление переводов
* Backend возвращает текст заявления в зависимости от языка интерфейса пользователя
* При языке интерфейса **ET** возвращается текст на эстонском
* При языке интерфейса **RU** возвращается русский перевод
* При языке интерфейса **EN** возвращается английский перевод
* Если перевод отсутствует — возвращается текст на **базовом языке (ET)**
* Логика корректно работает при создании, просмотре и согласовании заявления
* Реализация не требует создания отдельных заявлений для каждого языка
